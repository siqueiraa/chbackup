## AWS S3

The default configuration. Set the bucket, region, and credentials:

```yaml
s3:
  bucket: my-clickhouse-backups
  region: us-east-1
  access_key: AKIAIOSFODNN7EXAMPLE
  secret_key: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

If you are running on EC2 with an IAM instance role or on EKS with IRSA, you can omit `access_key` and `secret_key` entirely. The AWS SDK will pick up credentials from the environment automatically.

### IAM policy

The IAM user or role needs these permissions on the bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:ListBucket",
        "s3:GetBucketLocation",
        "s3:HeadObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": [
        "arn:aws:s3:::my-clickhouse-backups",
        "arn:aws:s3:::my-clickhouse-backups/*"
      ]
    }
  ]
}
```

If you use server-side CopyObject (for S3 disk tables or multipart copies), add `s3:GetObject` on the source bucket as well.

### S3 key prefix

All backup objects are stored under a configurable prefix. The default is `chbackup`:

```yaml
s3:
  prefix: chbackup
```

This produces keys like `chbackup/my-backup-name/metadata.json`. You can use ClickHouse macros in the prefix for multi-shard setups:

```yaml
s3:
  prefix: "chbackup/{shard}"
```

The `{shard}` macro is resolved from ClickHouse `system.macros` at runtime.

## MinIO

MinIO requires path-style addressing and a custom endpoint:

```yaml
s3:
  bucket: clickhouse-backups
  endpoint: "https://minio.example.com:9000"
  force_path_style: true
  access_key: minioadmin
  secret_key: minioadmin
```

For local development with HTTP (no TLS):

```yaml
s3:
  bucket: clickhouse-backups
  endpoint: "http://localhost:9000"
  force_path_style: true
  disable_ssl: true
  access_key: minioadmin
  secret_key: minioadmin
```

### Common MinIO issues

**"bucket does not exist"** -- Create the bucket before running chbackup:

```bash
mc mb myminio/clickhouse-backups
```

**TLS errors** -- If your MinIO uses a self-signed certificate, set `disable_cert_verification: true`. For production, use a proper certificate or `disable_ssl: true` behind a TLS-terminating proxy.

## Cloudflare R2

R2 uses S3-compatible API with an account-specific endpoint:

```yaml
s3:
  bucket: clickhouse-backups
  endpoint: "https://ACCOUNT_ID.r2.cloudflarestorage.com"
  region: auto
  force_path_style: true
  access_key: YOUR_R2_ACCESS_KEY
  secret_key: YOUR_R2_SECRET_KEY
```

Generate R2 API tokens in the Cloudflare dashboard under R2 > Manage R2 API Tokens. Select "Object Read & Write" permission.

R2 does not support all S3 features. These work: PutObject, GetObject, DeleteObject, ListObjectsV2, HeadObject, multipart upload. ACLs and some storage classes are not supported -- leave `acl` and `storage_class` empty.

## Ceph / RadosGW

Ceph with RadosGW exposes an S3-compatible API:

```yaml
s3:
  bucket: clickhouse-backups
  endpoint: "https://ceph-rgw.example.com"
  force_path_style: true
  access_key: YOUR_ACCESS_KEY
  secret_key: YOUR_SECRET_KEY
```

If your RadosGW does not support virtual-hosted-style requests (most do not), `force_path_style: true` is required.

## Authentication

chbackup supports three authentication methods, checked in this order:

### 1. Explicit credentials (config file or env vars)

```yaml
s3:
  access_key: AKIAIOSFODNN7EXAMPLE
  secret_key: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

Or via environment:

```bash
export S3_ACCESS_KEY=AKIAIOSFODNN7EXAMPLE
export S3_SECRET_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

### 2. IAM instance role (EC2)

Leave `access_key` and `secret_key` empty. The AWS SDK fetches credentials from the instance metadata service automatically.

### 3. IRSA (EKS)

On EKS with IAM Roles for Service Accounts, the SDK reads the `AWS_ROLE_ARN` and `AWS_WEB_IDENTITY_TOKEN_FILE` environment variables injected by the service account. No chbackup-specific configuration needed.

## Cross-account access (STS AssumeRole)

To write backups to a bucket in a different AWS account:

```yaml
s3:
  bucket: target-account-bucket
  region: us-east-1
  assume_role_arn: "arn:aws:iam::123456789012:role/chbackup-cross-account"
```

How it works:

1. chbackup creates an STS client using the base credentials (from config, env, or IAM role)
2. Calls `sts:AssumeRole` with session name "chbackup"
3. Uses the temporary credentials for all S3 operations

The assumed role needs the same S3 permissions listed in the [IAM policy](#iam-policy) section above.

**Credential expiry**: Temporary credentials from AssumeRole expire after 1 hour by default. In server mode, use `POST /api/v1/restart` to refresh them. In standalone CLI mode, credentials are fetched fresh on each invocation.

### Trust policy (target account)

The role in the target account needs a trust policy allowing the source account to assume it:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::SOURCE_ACCOUNT_ID:root"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

## Encryption

### SSE-S3 (AES-256)

Amazon manages the encryption keys. Simplest option:

```yaml
s3:
  sse: "AES256"
```

### SSE-KMS

You manage the key via AWS KMS:

```yaml
s3:
  sse: "aws:kms"
  sse_kms_key_id: "arn:aws:kms:us-east-1:123456789012:key/abcd-1234-efgh-5678"
```

The IAM user or role needs `kms:GenerateDataKey` and `kms:Decrypt` permissions on the KMS key.

### No encryption

Leave `sse` empty (the default). Objects are stored without server-side encryption. Bucket-level default encryption (if configured in AWS) still applies.

## Storage classes and ACLs

### Storage class

Controls the S3 storage tier for uploaded objects:

```yaml
s3:
  storage_class: STANDARD
```

Common values: `STANDARD`, `STANDARD_IA`, `ONEZONE_IA`, `GLACIER_IR`, `INTELLIGENT_TIERING`. The default is `STANDARD`.

Keep in mind that `GLACIER` and `DEEP_ARCHIVE` classes prevent direct download. You would need to initiate a restore in AWS before chbackup can download from these tiers.

### ACL

Set a canned ACL on uploaded objects:

```yaml
s3:
  acl: "bucket-owner-full-control"
```

Common values: `private`, `bucket-owner-full-control`. Leave empty to not set an ACL (the default). If the bucket has "Object Ownership" set to "Bucket owner enforced" (the AWS default for new buckets), ACLs are disabled and this setting is ignored.

## Performance tuning

### Upload concurrency

Controls how many parts are uploaded in parallel:

```yaml
general:
  upload_concurrency: 4     # parallel part uploads (default: 4)
  download_concurrency: 4   # parallel part downloads (default: 4)
```

Each "part" is one data part directory from ClickHouse, compressed independently. Higher values speed up backups with many small parts. Lower values reduce memory usage.

### Rate limiting

Throttle upload and download bandwidth (bytes per second):

```yaml
general:
  upload_max_bytes_per_second: 0     # 0 = unlimited
  download_max_bytes_per_second: 0   # 0 = unlimited
```

Useful for production servers where you do not want backups saturating the network.

### S3 disk object copy concurrency

When backing up tables stored on ClickHouse S3 disks, copies happen server-side via S3 CopyObject:

```yaml
general:
  object_disk_server_side_copy_concurrency: 32
```

This controls how many CopyObject calls run in parallel. Higher values help with large S3 disk tables.

### Multipart upload

Large parts (default > 256 MiB) use streaming multipart upload instead of buffering the entire part in memory:

```yaml
s3:
  max_parts_count: 10000   # maximum parts per multipart upload
  chunk_size: 0            # 0 = auto (object_size / max_parts_count)
```

Objects larger than 5 GB automatically use multipart CopyObject.

## Environment variables

All S3 config fields can be overridden via environment variables (`S3_BUCKET`, `S3_REGION`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_ENDPOINT`, etc.). Environment variables override config file values. CLI `--env` flags override both.

See [Configuration > S3 environment variables](configuration.md#s3) for the full list.

## Troubleshooting

### "Access Denied" errors

1. Verify credentials are correct: `aws s3 ls s3://your-bucket/ --region your-region`
2. Check that the IAM policy includes all required actions (see [IAM policy](#iam-policy))
3. If using AssumeRole, verify the trust policy on the target role
4. If the bucket uses a KMS key, check `kms:Decrypt` and `kms:GenerateDataKey` permissions

### "NoSuchBucket" or "bucket does not exist"

The bucket must exist before running chbackup. Create it manually:

```bash
aws s3 mb s3://my-clickhouse-backups --region us-east-1
```

### Slow uploads

- Increase `general.upload_concurrency` (default: 4)
- Check if rate limiting is set (`general.upload_max_bytes_per_second`)
- For tables with many small parts, the bottleneck is request overhead -- try compressing to `zstd` for better ratios
- Verify the S3 bucket is in the same region as your ClickHouse server

### "connection refused" with MinIO

- Check that `endpoint` includes the port number: `http://minio:9000`
- Check that `force_path_style: true` is set
- If running in Docker, use the container name or host network, not `localhost`

### Multipart upload failures

If a multipart upload fails partway through, chbackup aborts the upload. Incomplete multipart uploads do not consume storage. To clean up any lingering incomplete uploads in your bucket:

```bash
aws s3api list-multipart-uploads --bucket my-bucket
aws s3api abort-multipart-upload --bucket my-bucket --key KEY --upload-id ID
```

Or set a bucket lifecycle rule to auto-abort incomplete multipart uploads after N days.

### SSL/TLS errors

- **Self-signed certificate**: Set `s3.disable_cert_verification: true` (not recommended for production)
- **Wrong endpoint scheme**: If your S3 is HTTP-only, set `s3.disable_ssl: true` and use `http://` in the endpoint
- **Certificate chain issues**: Ensure `/etc/ssl/certs/` is populated (the Docker image includes `ca-certificates`)
