# Data Authority Analysis

## Data Requirements for Phase 1

| Data Needed | Source Type | Field Available | Decision |
|-------------|-------------|-----------------|----------|
| Table list for backup | ClickHouse `system.tables` | database, name, engine, create_table_query, uuid | USE EXISTING |
| Pending mutations | ClickHouse `system.mutations` | database, table, mutation_id, command, is_done, parts_to_do_names | USE EXISTING |
| Replica sync status | ClickHouse SQL | `SYSTEM SYNC REPLICA` is a command, not a query | USE EXISTING |
| Part checksums (CRC64) | Local filesystem | `checksums.txt` file inside each part directory | USE EXISTING — read from shadow/ after FREEZE |
| Part metadata (name, size) | Local filesystem | Part directory names follow `{partition}_{min}_{max}_{level}` convention; sizes from `std::fs::metadata` | USE EXISTING |
| ClickHouse version | ClickHouse `system.build_options` or `SELECT version()` | version string | USE EXISTING |
| Table DDL | ClickHouse `system.tables` | `create_table_query` field | USE EXISTING |
| Table UUID | ClickHouse `system.tables` | `uuid` field | USE EXISTING |
| Disk info | ClickHouse `system.disks` | name, path, type | USE EXISTING |
| Local backup directory listing | Local filesystem | Scan `/var/lib/clickhouse/backup/` | USE EXISTING |
| Remote backup listing | S3 `ListObjectsV2` | CommonPrefixes under `{prefix}/` | USE EXISTING |
| Compressed size tracking | Runtime calculation | Sum of bytes uploaded per part | MUST IMPLEMENT — no pre-existing source; computed during upload |
| Backup timestamp | Runtime | `chrono::Utc::now()` | MUST IMPLEMENT — generated at backup creation time |

## Analysis Notes

- **CRC64 of checksums.txt**: The design doc (section 3.4/3.5) specifies computing CRC64 of each part's `checksums.txt` file. The checksums.txt file is generated by ClickHouse and lives inside each data part. We read it from the shadow directory after FREEZE. This is a new computation we perform (not provided by ClickHouse), but the source data (the file) is existing.

- **Manifest**: The manifest is a new data structure we create. It aggregates data from multiple existing sources (table DDL, part info, checksums, timestamps). This is intentional design — it's the backup's self-contained metadata record.

- **Part names as identity**: The design doc confirms that part names are deterministic and serve as identity for incremental diffs. We do NOT need to track part IDs separately — the filesystem directory name IS the identifier.

- **Table engine type**: Available from `system.tables.engine`. Used for deciding ATTACH order (sequential for Replacing/Collapsing, parallel otherwise — but Phase 1 is sequential anyway).

- **File ownership (uid/gid)**: Detected by `stat()`-ing the ClickHouse data path. Uses `nix::unistd::chown()` — the `nix` crate is NOT in Cargo.toml yet and needs to be added.
